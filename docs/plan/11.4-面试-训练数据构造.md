# 11.4 - 面试问答：训练数据构造与质量评估

> 对应 JD 第 1 点："Coding Agent 数据构造 + SFT/DPO 训练"

---

## Q: "SFT 和 DPO 数据怎么构造的？"

### 面试怎么回答

> SFT 数据从成功轨迹中提取，格式是 (instruction, response) 对，response 是完整的 Thought→Action→Observation 链。DPO 需要偏好对比，我实现了两种配对策略：同任务的成功 vs 失败轨迹，以及都成功时步数少的 vs 步数多的。质量评估用效率评分，分为 good/usable/rejected 三级。

### 知识点详解

#### SFT（Supervised Fine-Tuning）数据格式

**定位代码**：`agent/trajectory.py:140-168` `export_sft_format()`

```json
{
  "instruction": "在 workspace/ 下创建 fizzbuzz.py...",
  "input": "",
  "output": "Thought: 我需要创建 fizzbuzz.py\nAction: write_file\nAction Input: {\"path\": \"fizzbuzz.py\", ...}\nObservation: Successfully wrote...\nThought: 现在写测试\n..."
}
```

**关键**：
- `instruction` = 原始任务描述
- `output` = **完整的 ReAct 轨迹链**（不是最终代码，而是思考→行动→观察的全过程）
- 只从**成功轨迹**中提取（失败轨迹不用于 SFT）
- 目的：让模型学会 Agent 的**推理和工具使用模式**，而不仅仅是写代码

#### DPO（Direct Preference Optimization）配对策略

**定位代码**：`agent/analyze.py` `generate_dpo_pairs()`

```json
{
  "prompt": "在 workspace/ 下创建 palindrome 函数...",
  "chosen": "Thought: ...(5步完成的高效轨迹)",
  "rejected": "Thought: ...(34步冗余的低效轨迹)",
  "pair_type": "efficient_vs_inefficient"
}
```

**两种配对方式**：

| 策略 | chosen | rejected | 适用场景 |
|------|--------|----------|---------|
| success_vs_failure | 成功轨迹 | 失败轨迹 | 同一任务，一个成功一个失败 |
| efficient_vs_inefficient | 步数最少的成功轨迹 | 步数最多的成功轨迹 | 同任务都成功，但效率不同 |

**DPO 的核心思想**：不告诉模型"正确答案是什么"，而是告诉它"A 比 B 好"。模型从对比中学会偏好更高效的解题路径。

#### 效率评分公式

**定位代码**：`agent/analyze.py` `analyze_single()`

```
efficiency = 1.0 - error_rate × 0.3 - redundancy_rate × 0.3 - step_penalty × 0.4

其中：
  error_rate = tool_errors / tool_calls           # 工具失败率
  redundancy_rate = redundant_calls / tool_calls   # 冗余调用率（连续相同工具+相同参数）
  step_penalty = max(0, (tool_calls - 10) / 20)   # 超过 10 步开始扣分
```

**权重设计理由**：
- 错误率 0.3：工具调用失败直接浪费 token 和时间
- 冗余率 0.3：重复调用说明 Agent 在兜圈子
- 步数 0.4：权重最高，因为步数直接决定成本和延迟

#### SFT 质量分级

| 等级 | 条件 | 说明 |
|------|------|------|
| **good** | 成功 + efficiency > 0.6 | 高质量，可直接用于 SFT |
| **usable** | 成功但 efficiency ≤ 0.6 | 可用但有噪声（步数多/冗余调用） |
| **rejected** | 失败 | 不用于 SFT，作为 DPO 的 rejected |

### 追问预判

| 追问 | 回答要点 |
|------|---------|
| "SFT 数据量够吗？" | 目前 5 条轨迹只是 Demo 验证。实际训练需要 1000+ 条，可以通过批量跑不同任务积累。重要的是数据质量（good 级别的比例）。 |
| "DPO 和 RLHF 有什么区别？" | RLHF 需要训练一个 reward model，然后用 PPO 优化。DPO 直接从偏好对比数据中学习，跳过了 reward model 这一步，更简单也更稳定。 |
| "为什么不用 RLHF？" | 数据规模小的时候 DPO 更实用。DPO 只需要偏好对，RLHF 需要大量标注数据训练 reward model。 |
| "冗余调用怎么检测的？" | 连续两次调用**相同工具 + 相同参数**就算冗余。比如连续两次 `read_file(path="main.py")` 没有意义。 |
| "效率评分公式怎么定的？" | 启发式设计，权重是根据实际 Demo 数据调的。核心原则：步数影响最大（0.4），因为每步都有 API 调用成本。 |

## Q: "轨迹数据的完整格式是什么？"

### 回答

**定位代码**：`agent/trajectory.py`

轨迹 JSON 文件结构：

```json
{
  "task": "创建 fizzbuzz.py ...",
  "start_time": "2026-03-21T15:37:05",
  "end_time": "2026-03-21T15:38:42",
  "success": true,
  "final_answer": "...",
  "steps": [
    {
      "agent": "coder",
      "step_type": "tool_call",
      "thought": "我需要创建 fizzbuzz.py",
      "action": "write_file",
      "action_input": {"path": "fizzbuzz.py", "content": "..."},
      "observation": "Successfully wrote to file: fizzbuzz.py",
      "is_error": false,
      "timestamp": "2026-03-21T15:37:10"
    },
    ...
  ]
}
```

**每个 step 的字段**对应 ReAct 的三个阶段：
- `thought` → Think（LLM 的推理）
- `action` + `action_input` → Act（工具调用）
- `observation` → Observe（工具返回）
