# 08 - Agent 效率优化

> 将平均步数从 64 步降至 16 步以内，SFT 质量从 usable 提升到 good

---

## 问题分析

Phase 3 的 4 条轨迹数据暴露了效率问题：

| 指标 | 目标 | 优化前 |
|------|------|--------|
| 平均步数 | ≤ 8 步 | 64 步 |
| 工具成功率 | ≥ 95% | 89.7% |
| 平均效率评分 | ≥ 0.6 | 35.3% |
| SFT 质量等级 | good | usable |

**根因**：Agent 倾向过度探索（反复 `list_files` + `code_search` + `read_file`），System Prompt 不够 directive。

---

## 优化措施

### 1. 降低工具调用上限（config.py）

```python
# 之前
MAX_TOOL_CALLS = 15     # 所有角色统一 15 次

# 之后：按角色差异化配置
MAX_TOOL_CALLS = 8            # Coder: 8 次（核心编码不超过 8 步）
MAX_TOOL_CALLS_REVIEWER = 3   # Reviewer: 读代码 + 给结论
MAX_TOOL_CALLS_TESTER = 5     # Tester: 跑测试 + 验证
```

**总上限**：单轮 Coder(8) + Reviewer(3) + Tester(5) = 16 步，3 轮最多 48 步。

### 2. Coder Prompt 精简（coder.py）

**优化前**：鼓励先 list_files、code_search 理解项目结构
**优化后**：强调 3-5 步完成，列出反模式

```
Typical flow (3-5 tool calls):
1. read_file (if modifying existing code)
2. write_file (create/modify)
3. execute_code (verify it works)
4. Done

Anti-patterns to AVOID:
- Do NOT call list_files unless task requires finding unknown files
- Do NOT read files you just wrote
- Do NOT call same tool twice with same arguments
```

### 3. Reviewer Prompt 精简（reviewer.py）

**优化前**：4 个检查维度 + 3 步工作流
**优化后**：1-2 次工具调用，只关注正确性 bug

```
Process (1-2 tool calls max):
1. read_file to see the actual code
2. Give verdict immediately

Rules:
- Focus ONLY on correctness bugs. Ignore style.
- If code works correctly, say LGTM.
```

### 4. Tester Prompt 精简（tester.py）

**优化前**：先读文件、再跑测试、再分析
**优化后**：直接跑测试，通过就说 PASS

```
Process (2-3 tool calls max):
1. run_tests OR execute_code
2. Give verdict immediately

Rules:
- Run tests ONCE. If pass, say PASS immediately.
- Do NOT re-run tests that already passed.
```

### 5. 后续轮次反馈精简（coder.py）

**优化前**：每次重试都带完整任务描述 + 全部反馈
**优化后**：只给需要修的点，要求最小化修复

```python
# 优化后
revision_prompt = f"Fix the following issues (iteration {iteration}):\n"
revision_prompt += f"\nTest failure:\n{test_feedback}\n"
revision_prompt += "\nMake the minimal fix needed. Do not rewrite the whole file."
```

## 实际验证结果

### Demo 5: FizzBuzz（优化后首次验证）

| 指标 | 优化前 (Demo 1) | 优化后 (Demo 5) | 改进 |
|------|----------------|----------------|------|
| 总步数 | 34 步 | **14 步** | **-59%** |
| 工具调用 | ~30 | **11** | **-63%** |
| Coder 步数 | ~25 | **7** (含 1 次 ls 冗余) | -72% |
| Reviewer 步数 | ~5 | **3** (读 2 个文件 + 结论) | -40% |
| Tester 步数 | ~5 | **4** (run_tests + execute_code + read_file) | -20% |
| 工具成功率 | 87% | **100%** | +13% |
| 结果 | PASS | **PASS** | - |
| 迭代轮次 | 1 轮 | **1 轮** | - |

**分析**：
- Coder 遵循了 3-5 步的建议流程：write → write → execute → execute → ls → done
- 还有一步冗余（`ls -la *.py`），但整体已大幅改善
- Reviewer 只读了 2 个文件就给出 LGTM
- Tester 的 `read_file` 是冗余的（已经测试通过了），后续可进一步优化
