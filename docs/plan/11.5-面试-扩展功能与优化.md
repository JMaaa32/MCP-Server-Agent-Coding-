## Q: "RAG 代码检索是怎么实现的？"

### 面试怎么回答

> 实现了一个轻量级的结构化代码检索引擎。先把代码按语义单元（函数、类）切分成 chunk，然后用 TF-IDF 向量化。查询时通过余弦相似度排序，返回最相关的代码片段。和 code_search 的关键词匹配不同，RAG 支持模糊语义检索——搜"排序算法"能匹配到 `sort_list` 函数。

### 知识点详解

#### 整体架构

**定位代码**：`agent/rag.py`

```
输入: workspace/ 下的源文件
       ↓
代码切片 (按函数/类边界)
       ↓
TF-IDF 向量化 (驼峰/下划线分词)
       ↓
存储: list[CodeChunk]，每个带 vector
       ↓
查询: query text → 向量化 → 余弦相似度 → top-k
```

#### 代码切片策略

| 语言 | 切片方式 | 切片粒度 |
|------|---------|---------|
| Python | 正则匹配 `def`/`class` + 缩进边界 | 每个函数/类一个 chunk |
| C/C++ | 正则匹配函数/class + 大括号匹配 | 每个函数/类一个 chunk |
| 其他 | 固定窗口（30 行，步长 20 行） | 滑动窗口 |

**为什么按函数切而不是按行切**：
- 按行切会把一个函数拆成多个片段，失去语义完整性
- 按函数切，检索返回的是完整的代码单元，Agent 可以直接理解和使用

#### TF-IDF 向量化

**定位代码**：`agent/rag.py` `TFIDFVectorizer`

```
分词: "parseJSONData" → ["parse", "json", "data"]  (驼峰拆分)
      "sort_list"     → ["sort", "list"]            (下划线拆分)

TF = 词频 / 总词数
IDF = log(文档总数 / 包含该词的文档数 + 1) + 1
TF-IDF = TF × IDF

向量: [tf_idf_word1, tf_idf_word2, ..., tf_idf_wordN]  (N ≤ 5000)
```

**为什么不用 Embedding API**：
- TF-IDF 纯 Python 实现，无外部依赖，无 API 调用成本
- 对代码检索来说，TF-IDF 的标识符匹配已经够用（代码不像自然语言那样有大量同义词）
- 后续可以替换成 DeepSeek Embedding API 作为升级

#### 余弦相似度

```
cosine(a, b) = (a · b) / (|a| × |b|)
```

值域 [0, 1]，1 表示完全相同，0 表示无关。

#### 实际效果

```bash
$ python3 rag.py ../workspace "sort"

[0.7764] tasks/test_buggy.py:6-7  test_sort_basic
[0.7544] tasks/test_buggy.py:14-15 test_sort_reverse
[0.6669] tasks/test_buggy.py:10-11 test_sort_already_sorted
[0.2518] tasks/buggy.py:4-14      sort_list
```

34 个 chunks 索引了 9 个文件，检索结果合理。

### 追问预判

| 追问 | 回答要点 |
|------|---------|
| "TF-IDF 和 Embedding 的区别？" | TF-IDF 是稀疏向量（基于词频统计），Embedding 是稠密向量（神经网络学到的语义表示）。TF-IDF 适合精确词匹配，Embedding 适合语义相似（如 "sort" 和 "排序"）。 |
| "代码检索和自然语言检索有什么区别？" | 代码有明确的结构（函数、类），可以用 AST/正则精确切分。代码标识符是自造词（`parseJSON`），需要驼峰拆分才能检索。自然语言的 chunk 边界不确定（段落？句子？）。 |
| "为什么 vocab 限制 5000？" | 内存和性能考虑。代码的词汇量远小于自然语言，5000 词覆盖大部分标识符和关键词。更大的 vocab 增加向量维度但收益递减。 |
| "和 code_search 有什么区别？" | code_search 是 C++ 实现的精确匹配（关键词/正则/符号），需要知道确切名称。RAG 支持模糊查询（"排序" 能匹配 sort_list），更适合 Agent 在不确定要找什么时使用。 |

## Q: "analyze_code 的复杂度分析怎么做的？"

### 面试怎么回答

> 用静态分析估算代码复杂度。核心是圈复杂度（Cyclomatic Complexity），通过计算分支语句数量得出。还分析嵌套深度、注释率、函数列表和导入依赖。

### 知识点详解

#### 圈复杂度（Cyclomatic Complexity）

**定位代码**：`src/server/builtin/tools/analyze_code_tool.cpp`

Thomas McCabe 1976 年提出的经典方法：

```
CC = 1 + count(decision_points)

Python: if, elif, for, while, except, and, or
C++:    if, else if, for, while, switch, case, catch
```

**评级**：

| CC 值 | 评级 | 含义 |
|-------|------|------|
| 1-5 | low | 简单函数 |
| 6-15 | moderate | 中等复杂，需要注意 |
| 16-30 | high | 复杂，应该考虑重构 |
| >30 | very_high | 过于复杂，维护困难 |

**原理**：每个分支语句增加一条独立执行路径，路径越多意味着测试和理解的难度越大。

#### 嵌套深度

- Python：缩进级别 / 4（因为 Python 标准缩进是 4 空格）
- C++：追踪 `{` `}` 的嵌套深度

嵌套深度和可读性直接相关，超过 4 层一般建议提取函数。

### 追问预判

| 追问 | 回答要点 |
|------|---------|
| "这个和 Tree-sitter 的区别？" | Tree-sitter 生成完整的 AST，解析精确但需要额外依赖。我用正则匹配，轻量级（零依赖），精确度够用。trade-off 是不处理复杂嵌套/宏定义。 |
| "圈复杂度有什么局限？" | 不区分分支的复杂度（一个简单的 if 和一个复杂的 if 算同一个决策点）。也不考虑代码的认知复杂度（如回调嵌套）。认知复杂度（Cognitive Complexity）是更现代的指标。 |

## Q: "你怎么优化 Agent 效率的？"

### 面试怎么回答

> 跑了 4 条轨迹后发现平均 64 步，远超目标。分析发现 Agent 过度探索——反复 list_files、code_search。优化三方面：差异化工具上限、Prompt 工程加反模式约束、精简后续轮次反馈。结果步数从 34 降到 14，改善 59%。

### 知识点详解

#### 根因分析

原始 Prompt 写了："Use `list_files` or `code_search` to understand the project structure"

→ Agent 每次任务第一件事就是花 5-10 步探索项目结构

→ 即使任务描述已经告诉了它要操作哪些文件

#### 三个优化措施

**1. 差异化工具上限**

```python
# 之前：一刀切
MAX_TOOL_CALLS = 15  # 所有角色

# 之后：按角色
MAX_TOOL_CALLS = 8            # Coder（核心编码）
MAX_TOOL_CALLS_REVIEWER = 3   # Reviewer（读代码给结论）
MAX_TOOL_CALLS_TESTER = 5     # Tester（跑测试验证）
```

**2. Prompt 加反模式约束**

```
## Anti-patterns to AVOID:
- Do NOT call list_files unless the task requires finding unknown files
- Do NOT read files you just wrote
- Do NOT call same tool twice with same arguments
- Do NOT explore project structure when task tells you exactly what to do
```

**3. 后续轮次精简反馈**

```python
# 之前：重复完整任务描述 + 所有反馈
revision_prompt = f"Task: {task}\n\nReviewer feedback:\n{feedback}\n\nTest results:\n{test_feedback}"

# 之后：只给要修的点
revision_prompt = f"Fix the following issues (iteration {iteration}):\n"
revision_prompt += f"\nTest failure:\n{test_feedback}\n"
revision_prompt += "\nMake the minimal fix needed. Do not rewrite the whole file."
```

#### 效果

| 指标 | 优化前 (Demo 1) | 优化后 (Demo 5) | 改善 |
|------|----------------|----------------|------|
| 总步数 | 34 | **14** | -59% |
| 工具调用 | ~30 | **11** | -63% |
| 工具成功率 | 87% | **100%** | +13% |

### 追问预判

| 追问 | 回答要点 |
|------|---------|
| "上限设太低会不会导致任务失败？" | 可能。但更倾向于让 Agent 学会在限制内高效完成，而不是给它无限的探索空间。如果 8 步确实不够，会触发 retry（回到 Coder 重做），等于有兜底。 |
| "Prompt 工程的局限是什么？" | 依赖模型的指令遵循能力。DeepSeek-chat 遵循得比较好，但小模型（如 qwen3:8b）可能忽略反模式约束。这也是为什么还要加硬性的上限。 |
| "还能怎么优化？" | (1) 上下文压缩：工具返回结果太长时截断/摘要。(2) 工具选择优化：给 Agent 推荐最可能用到的工具。(3) 缓存：相同工具+相同参数的调用直接返回缓存结果。 |

## Q: "SSE 实时推送怎么做的？"

### 面试怎么回答

> SSE 是 HTTP 的单向推送协议，比 WebSocket 简单。我在 Agent 主进程里启动后台线程跑 HTTP 服务器，Agent 执行工具调用时通过全局单例 push 事件。浏览器 `new EventSource('/events')` 就能实时看到 Agent 状态。

### 知识点详解

**定位代码**：`agent/sse.py`

#### SSE vs WebSocket

| 特性 | SSE | WebSocket |
|------|-----|-----------|
| 方向 | 服务器 → 客户端（单向） | 双向 |
| 协议 | HTTP | 独立协议（ws://） |
| 重连 | 浏览器自动重连 | 需要手动实现 |
| 复杂度 | 简单（普通 HTTP 响应） | 复杂（握手升级） |
| 适用场景 | 日志推送、状态监控 | 聊天、实时协作 |

Agent 执行状态只需要**服务器推给客户端**，不需要客户端发消息给服务器，所以 SSE 更合适。

#### 实现架构

```python
SSEServer（后台 daemon 线程）
├── HTTPServer 监听 0.0.0.0:8765
├── /events → SSE 事件流（text/event-stream）
├── /status → JSON 状态查询
└── /       → 内置 HTML 监控页

push() → 遍历所有客户端队列 → put(event)
```

#### 关键代码

```python
# 推送事件（在 Agent 节点中调用）
sse_server.push("tool_call", {"agent": "coder", "tool": "write_file", "args": {...}})

# SSE 响应格式
"event: tool_call\ndata: {\"agent\":\"coder\",...}\n\n"

# 心跳保活（30 秒无事件时）
": heartbeat\n\n"
```

### 追问预判

| 追问 | 回答要点 |
|------|---------|
| "多个客户端怎么处理？" | 每个客户端连接时创建独立的 Queue，push() 遍历所有 Queue 广播。客户端断开时从列表移除。 |
| "线程安全怎么保证？" | 客户端列表用 `threading.Lock` 保护，Queue 本身是线程安全的。 |
| "为什么用后台线程而不是单独进程？" | Agent 执行和 SSE 需要共享内存（全局单例 sse_server），线程天然共享地址空间，进程需要 IPC。 |
