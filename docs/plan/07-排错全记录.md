## 坑 1：macOS 编译报错 — `std::filesystem::file_time_type` 无法转换

### 报错现场

```
list_files_tool.cpp:XX:XX: error: no matching function for call to 'duration_cast'
  auto mtime = std::chrono::duration_cast<std::chrono::seconds>(
               entry.last_write_time().time_since_epoch()).count();
```

### 背景

`list_files_tool.cpp` 需要返回文件修改时间。Linux 下 `std::filesystem::file_time_type` 的 clock 是 `system_clock`，可以直接 `duration_cast`。但 macOS (Apple Clang) 用的是一个内部的 `__file_clock`，不兼容 `system_clock`，无法直接转换。

### 排查过程

1. 一开始直接写了标准 C++17 的 `last_write_time().time_since_epoch().count()` — 编译报错
2. 搜索发现是 Apple Clang 的已知问题，`file_time_type` 用的是非标准 clock
3. C++20 有 `clock_cast` 可以解决，但项目用的是 C++17

### 修复

改用 POSIX `stat()` 系统调用直接获取 `st_mtime`：

```cpp
// list_files_tool.cpp:118-121
#include <sys/stat.h>

struct stat st;
if (::stat(entry.path().c_str(), &st) == 0) {
    file_info["modified_time"] = static_cast<int64_t>(st.st_mtime);
}
```

### 定位代码

`src/server/builtin/tools/list_files_tool.cpp:7`（`#include <sys/stat.h>`）和 `:118-121`

### 面试话术

> macOS 的 Apple Clang 对 C++17 filesystem 支持不完整，`file_time_type` 用的是非标准的 `__file_clock`，不能直接转 `system_clock`。C++20 才有 `clock_cast` 解决这个跨 clock 转换问题。既然我们项目锁定 C++17，就用了 POSIX `stat()` 直接拿 `st_mtime`，兼容性最好。

## 坑 2：MCP Server 启动 JSON 解析崩溃

### 报错现场

```
$ ./build/src/mcp_server --port 8089
[json.exception.parse_error.101] parse error at line 1, column 1: ...
```

Server 直接崩溃退出，没有任何有用信息。

### 排查过程

1. 第一反应是端口冲突，但 `lsof -i :8089` 没有其他进程
2. 看了 `mcp_server_main.cpp`，发现 main 函数里先加载 `--config` 指定的配置文件
3. 不传 `--config` 时，代码尝试读取默认路径的 JSON 配置，文件不存在或为空 → `nlohmann::json::parse()` 抛异常

### 修复

启动时必须带 `--config`：

```bash
./build/src/mcp_server --config config/server.json --port 8089
```

### 面试话术

> 这是一个防御性编程的问题。JSON 配置加载没有做文件存在性检查，`nlohmann::json::parse()` 对空输入直接抛异常。正确做法是加一个文件检查，不存在时用默认配置。

## 坑 3：Ollama 调用持续 502 — 系统代理劫持 localhost

### 报错现场

```
httpx.HTTPStatusError: Server error '502 Bad Gateway' for url 'http://localhost:11434/api/chat'
```

每次调用 Ollama 都 502，无论简单还是复杂的请求。

### 背景

系统设置了 HTTP 代理：`HTTP_PROXY=http://127.0.0.1:7890`（Clash/V2Ray）。Ollama Python SDK 内部用的是 `httpx`，`httpx` 会读取系统代理环境变量。

### 排查过程（共 4 次尝试）

**尝试 1：加 `think=False` 参数**
- 以为是 Ollama API 参数问题 → 依然 502，无效

**尝试 2：在代码中清除代理环境变量**
```python
os.environ.pop("HTTP_PROXY", None)
os.environ.pop("HTTPS_PROXY", None)
```
- `httpx` 可能在导入时就缓存了代理设置 → 依然 502，无效

**尝试 3：传入自定义 `httpx.Client`**
```python
import httpx
client = httpx.Client(proxies={})
ollama.chat(client=client, ...)
```
- `ollama` SDK 0.6.1 版本不支持 `_client` 参数 → TypeError，无效

**尝试 4（最终方案）：弃用 ollama SDK，直接 requests**
```python
resp = requests.post(
    "http://localhost:11434/api/chat",
    json=payload,
    proxies={"http": "", "https": ""},  # 关键：空字符串 = 不用代理
    timeout=300,
)
```
- ✅ 成功！`requests` 库的 `proxies={"http": ""}` 会明确绕过代理

### 定位代码

`agent/llm.py:156-189` — `_chat_ollama()` 函数，注释写了 "绕过 httpx 的 502 问题"

### 关键知识点

| 库 | 代理行为 | 绕过方式 |
|----|---------|---------|
| `httpx` | 自动读 `HTTP_PROXY` 环境变量，且缓存 | 传 `proxies={}` 给 Client 构造函数 |
| `requests` | 自动读环境变量，但可以按请求覆盖 | `proxies={"http": "", "https": ""}` |
| `curl` | 自动读环境变量 | `--noproxy localhost` 或 `curl -x ""` |

### 面试话术

> 这个问题花了最多时间排查。表面是 502，但 Ollama 本身是正常的（`curl --noproxy localhost` 可以调通）。根因是 `ollama` Python SDK 用的 `httpx` 库会走系统代理，代理不理解 Ollama 的 API 就返回 502。尝试了清环境变量、传自定义 client，都不行（httpx 行为和 SDK 版本限制）。最终放弃 SDK，用 `requests.post()` 直接调 REST API，通过 `proxies={"http": ""}` 明确不走代理。这个经验也应用到了 `mcp_client.py`——所有 localhost 请求都加了 `proxies={"http": "", "https": ""}`。

## 坑 5：DeepSeek API 400 — tool_call_id 格式不对

### 报错现场

```
openai.BadRequestError: Error code: 400 - {'error': {'message': 'invalid request: ...', 'type': 'invalid_request_error'}}
```

DeepSeek 拒绝了我们回传的工具结果消息。

### 背景

DeepSeek API 兼容 OpenAI 格式，但对 tool calling 的消息格式要求比较严格：

1. assistant 消息的 `tool_calls` 数组中每个 tool_call 必须有 `id`、`type: "function"`
2. `arguments` 必须是 **JSON 字符串**（不是 dict 对象）
3. tool result 消息的 `tool_call_id` 必须匹配

### 排查过程

1. 第一版直接把 Ollama 格式发给 DeepSeek → 400
2. 对比 OpenAI 文档，发现缺了 `"type": "function"` 字段
3. 修了 type 字段后还是 400 — 因为 `arguments` 传的是 dict，DeepSeek 要求 JSON string
4. 加了 `json.dumps(arguments)` 后终于通过

### 修复

```python
# llm.py:70-78 — make_tool_result_messages() 中
hist_msg["tool_calls"].append({
    "id": tc.get("id", _next_call_id()),
    "type": "function",                    # 必须有
    "function": {
        "name": tc["function"]["name"],
        "arguments": json.dumps(tc["function"]["arguments"]),  # 必须是字符串
    }
})
```

### 定位代码

`agent/llm.py:62-88` — `make_tool_result_messages()` 的 DeepSeek 分支

### 面试话术

> DeepSeek 虽然说兼容 OpenAI，但 tool calling 的格式要求比 OpenAI 更严格。调了半天 400 错误，发现两个问题：一是 assistant 消息的 tool_calls 缺了 `"type": "function"` 字段，二是 `arguments` 要求是序列化后的 JSON 字符串而不是 dict 对象。这就是为什么我在 `make_tool_result_messages()` 里做了专门的格式转换，DeepSeek 和 Ollama 走不同的分支。

## 坑 6：execute_code 分号过滤误杀合法 Python 命令

### 报错现场

Agent 调用 `execute_code` 执行 `python3 -c "import os; print(os.getcwd())"` 时返回：

```json
{"error": "Command contains dangerous patterns: ;"}
```

### 背景

`execute_code_tool.cpp` 有一层危险符号检测，原版直接在整个命令字符串里搜索 `;`、`&&`、`|` 等。但 `python3 -c "a=1; b=2; print(a+b)"` 中的分号是 Python 语句分隔符，在引号内，并不是 shell 命令拼接。

### 排查过程

1. Agent 的 Demo 1 中出现大量工具失败（工具成功率只有 87%），多数是 execute_code 被拒
2. 查看轨迹数据，发现 Agent 经常用 `python3 -c "code;code"` 的形式执行代码
3. 分析：引号内的 `;` 对 shell 是安全的（不会被解释为命令分隔符）
4. 设计方案：只检查引号**外**的危险符号

### 修复

新增 `extract_unquoted_parts()` 函数：

```cpp
// execute_code_tool.cpp:26-55
static std::string extract_unquoted_parts(const std::string& cmd) {
    std::string result;
    bool in_single = false, in_double = false, escaped = false;
    for (size_t i = 0; i < cmd.size(); ++i) {
        char c = cmd[i];
        if (escaped) { escaped = false; continue; }
        if (c == '\\') { escaped = true; continue; }
        if (c == '\'' && !in_double) { in_single = !in_single; continue; }
        if (c == '"' && !in_single) { in_double = !in_double; continue; }
        if (!in_single && !in_double) { result += c; }
    }
    return result;
}
```

`contains_dangerous_patterns()` 改为检查 `extract_unquoted_parts(cmd)` 而不是原始 `cmd`。

### 验证

```
✅ python3 -c "import os; print(os.getcwd())"  → 允许（分号在引号内）
✅ python3 -c 'a=1; b=2; print(a+b)'           → 允许（分号在引号内）
❌ python3 test.py; rm -rf /                    → 拦截（分号在引号外）
❌ python3 test.py && cat /etc/passwd            → 拦截（&& 在引号外）
```

### 定位代码

`src/server/builtin/tools/execute_code_tool.cpp:26-55`（`extract_unquoted_parts`）和 `:58-72`（`contains_dangerous_patterns`）

### 效果

Demo 4 工具成功率从 87% 提升到 95%。

### 面试话术

> 安全检查不能太粗暴。原来对整个命令字符串做子串匹配，结果 `python3 -c "a;b"` 被误杀。shell 的实际行为是：引号内的字符不会被 shell 解释，所以分号只在引号外才是命令分隔符。我实现了一个引号感知的解析器 `extract_unquoted_parts()`，追踪单引号、双引号和转义状态，只提取引号外的部分来检查危险符号。这是一个状态机的经典应用。

## 坑 7：write_file 没有沙箱限制 — 路径穿越漏洞

### 发现过程

`read_file_tool.cpp` 一开始就有 `weakly_canonical()` + 前缀检查的沙箱逻辑。但 Code Review 时发现 `write_file_tool.cpp` **没有**做路径验证——Agent 可以写任意文件。

这意味着如果 LLM 被 prompt injection 攻击（比如读到一个包含恶意指令的文件），它可以调用 `write_file(path="../../etc/crontab", content="...")` 来写入系统文件。

### 修复

给 `write_file_tool.cpp` 补上和 `read_file_tool.cpp` 一样的逻辑：

```cpp
// write_file_tool.cpp:35-50
fs::path target = raw_path;
if (target.is_relative()) {
    target = sandbox_root / target;
}
std::error_code ec;
fs::path canonical_path = fs::weakly_canonical(target, ec);  // weakly 因为目标文件可能不存在
if (ec) { return error; }

auto sandbox_str = sandbox_root.string();
if (canonical_path.string().substr(0, sandbox_str.size()) != sandbox_str) {
    MCP_LOG_WARN("write_file: path traversal blocked: {}", raw_path);
    return error;
}
```

### 为什么用 `weakly_canonical` 而不是 `canonical`

- `canonical()` 要求路径上所有组件必须**实际存在**，不存在就抛异常
- `weakly_canonical()` 会解析已存在的部分，不存在的部分保留原样
- write_file 的目标文件通常还不存在（创建新文件），所以必须用 `weakly_canonical`

### 定位代码

`src/server/builtin/tools/write_file_tool.cpp:35-50`

### 面试话术

> 发现 write_file 缺少沙箱验证是在做 Code Review 的时候。read_file 有路径检查但 write_file 没有，这是个路径穿越漏洞。修复方案和 read_file 一样：`weakly_canonical()` 规范化路径后检查前缀。注意用的是 `weakly_canonical` 而不是 `canonical`——因为写文件时目标文件还不存在，`canonical` 要求路径实际存在会直接报错。

## 坑 8：DeepSeek API 间歇性连接失败

### 报错现场

```
openai.APIConnectionError: Connection error.
```

不是每次都失败，跑长任务时（20+ 步）偶尔出现。

### 排查过程

1. 不是网络断了（curl 手动测试正常）
2. 可能是 OpenAI SDK 的连接池问题，或者 DeepSeek 服务端偶尔限流
3. 加了重试后稳定了

### 修复

```python
# llm.py:103-108
client = OpenAI(
    api_key=config.DEEPSEEK_API_KEY,
    base_url=config.DEEPSEEK_BASE_URL,
    timeout=60.0,       # 从默认值提高
    max_retries=2,      # SDK 层面自动重试
)

# llm.py:118-129 — 业务层面再加一层重试
max_retries = 3
for attempt in range(max_retries):
    try:
        response = client.chat.completions.create(**kwargs)
        break
    except Exception as e:
        if attempt < max_retries - 1:
            wait = 2 ** attempt     # 1s, 2s 指数退避
            time.sleep(wait)
        else:
            raise
```

### 面试话术

> DeepSeek API 偶尔会连接失败，不确定是服务端限流还是网络波动。解决方案是双层重试：SDK 层面 `max_retries=2`（处理底层 HTTP 重试），业务层面再套一层 3 次重试加指数退避。这种模式在调用任何外部 API 时都建议加上。

## 坑 9：run_tests 用 pytest 但 sandbox 环境没装

### 报错现场

Agent 调用 `run_tests` 工具：

```json
{
  "exit_code": 1,
  "output": "/Applications/Xcode.app/.../python3: No module named pytest\n",
  "all_passed": false
}
```

### 背景

`run_tests_tool.cpp:101-104` 对 Python 测试文件固定使用 `python3 -m pytest` 执行。但 macOS 系统自带的 Python3（Xcode CommandLineTools）没有安装 pytest。

### 排查过程

1. Demo 5 中 Coder 写了 `test_fizzbuzz.py`，使用的是 `unittest` 框架
2. Tester Agent 调用 `run_tests`，底层执行 `python3 -m pytest test_fizzbuzz.py` → 失败
3. 但 Tester 足够聪明——看到 pytest 不可用后，主动切换到 `execute_code` 执行 `python3 test_fizzbuzz.py`，成功了

### 现状

这个坑没有在 C++ 层修复（`run_tests` 仍然硬编码 pytest），但 Agent 通过自主 fallback 解决了。这其实是 ReAct 模式的一个优势——工具调用失败后 LLM 能自主调整策略。

### 如果要修

方案一：`run_tests_tool.cpp` 先检测 `python3 -m pytest --version`，失败则回退到 `python3 <file>`
方案二：在 `run_tests` 的 description 中说明需要 pytest，让 Agent 自己选择用 `execute_code` 还是 `run_tests`

### 面试话术

> run_tests 硬编码了 pytest 但环境里没装。有趣的是 Agent 自己解决了这个问题——Tester 看到 pytest 失败后自动改用 `execute_code` 执行 `python3 test_fizzbuzz.py`。这体现了 ReAct 模式的鲁棒性：工具失败不等于任务失败，LLM 能根据错误信息自主调整策略。

## 坑 10：Agent 步数过多 — Prompt 引导不当

### 现象

| Demo | 步数 | 分析 |
|------|------|------|
| Demo 1 (palindrome) | 34 步 | Coder 反复 list_files + code_search 探索 |
| Demo 3 (code understanding) | 143 步 | 3 轮迭代都无法满足 Tester |
| Demo 4 (fibonacci) | 46 步 | Coder 步数合理，但 Reviewer/Tester 冗余 |

### 根因分析

1. **Coder Prompt 鼓励探索**：原 Prompt 写了 "Use `list_files` or `code_search` to understand the project structure"，Agent 每次都先花 5-10 步探索
2. **Reviewer 太仔细**：原 Prompt 有 4 个检查维度（correctness, edge cases, code quality, potential bugs），导致 Reviewer 反复读文件、搜索代码
3. **Tester 冗余验证**：测试通过后还会 read_file 看代码，不直接给 PASS
4. **MAX_TOOL_CALLS = 15 太宽松**：单个节点 15 步 × 3 个节点 = 45 步/轮

### 修复（详见 08-efficiency-optimization.md）

```python
# config.py — 差异化上限
MAX_TOOL_CALLS = 8           # Coder
MAX_TOOL_CALLS_REVIEWER = 3  # Reviewer
MAX_TOOL_CALLS_TESTER = 5    # Tester

# coder.py — 精简 Prompt
"""
Typical flow (3-5 tool calls):
1. read_file (if modifying existing code)
2. write_file
3. execute_code (verify)
4. Done

Anti-patterns to AVOID:
- Do NOT call list_files unless the task requires finding unknown files
- Do NOT read files you just wrote
"""
```

### 效果

Demo 5 (fizzbuzz)：34 步 → **14 步**，改善 59%。

### 面试话术

> Prompt 工程对 Agent 效率影响极大。原来的 Prompt 鼓励 Agent 先理解项目结构再动手，结果它每次花 10 步去 list_files + code_search。改成给出 3-5 步的典型流程并明确列出反模式后，步数直接从 34 降到 14。另外给 Reviewer 和 Tester 设了更低的工具调用上限（3 次和 5 次），避免它们过度验证。

- "所有 .py 文件" — 范围不确定
- "API 文档" — 没有明确的输出格式/内容规范
- Tester 不知道什么算"完整"，每次都找新的理由 FAIL

### 教训

给 Agent 的任务描述需要：
1. **明确输入**：具体哪些文件/功能
2. **明确输出**：文件名、格式、必须包含的内容
3. **明确验证标准**：怎样算通过

好的任务描述示例：
```
在 workspace/ 下创建 fizzbuzz.py：实现 fizzbuzz(n) 函数，返回 1 到 n 的 FizzBuzz 列表。
写 test_fizzbuzz.py 测试，覆盖 n=1, 15, 0 三种情况。
```

### 面试话术

> 这不是代码问题，是 Agent 工程问题。Task description 的质量直接决定了 Agent 的成功率。开放式任务（"分析所有文件生成文档"）没有明确的验证标准，Tester 每轮都能找到新理由 FAIL，导致 3 轮迭代都无法收敛。后来我改成给具体的输入输出规范后，一轮就通过了。这也是为什么 Coding Agent 更适合做结构化的、有明确完成标准的任务。

## 复现指南

如果你想自己走一遍完整流程，按以下顺序：

### 1. 编译 C++ MCP Server

```bash
cd mcp-tutorial
cmake -B build -S . -DCMAKE_TOOLCHAIN_FILE=~/vcpkg/scripts/buildsystems/vcpkg.cmake
cmake --build build
```

如果遇到坑 1（file_clock），检查 `list_files_tool.cpp` 是否用了 `stat()`。

### 2. 启动 Server

```bash
./build/src/mcp_server --config config/server.json --port 8089
```

如果遇到坑 2（JSON 解析崩溃），确认 `--config` 参数正确。

### 3. 验证 Server

```bash
curl --noproxy localhost http://localhost:8089/jsonrpc \
  -X POST -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","id":1,"method":"tools/list"}'
```

如果返回空或 502，是坑 11（代理劫持），加 `--noproxy localhost`。

### 4. 安装 Python 依赖

```bash
cd agent
pip3 install openai requests rich langgraph
```

### 5. 配置环境变量

```bash
export DEEPSEEK_API_KEY="your-key-here"
export LLM_BACKEND="deepseek"
```

### 6. 跑 Demo

```bash
python3 main.py "在 workspace/ 下创建 fizzbuzz.py：实现 fizzbuzz(n) 函数，返回 1 到 n 的 FizzBuzz 列表。写 test_fizzbuzz.py 测试。"
```

如果遇到坑 5（DeepSeek 400），检查 `llm.py` 的 `make_tool_result_messages()`。
如果遇到坑 6（分号误杀），检查 `execute_code_tool.cpp` 是否有 `extract_unquoted_parts()`。

### 7. 分析轨迹

```bash
python3 analyze.py --report --export-dpo --save-report
```
