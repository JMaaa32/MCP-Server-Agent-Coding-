## Q: "LangGraph 状态图怎么工作的？"

### 回答

**定位代码**：`agent/graph.py:44-103`

```
状态图结构:
  START → coder → reviewer → tester → should_continue
                                          │
                                          ├─ "done"        → END（测试通过）
                                          ├─ "retry"       → coder（失败，重做）
                                          └─ "max_reached" → END（3 轮上限）
```

**核心代码**：

```python
# graph.py:78-101
graph = StateGraph(CodingAgentState)

graph.add_node("coder", _coder)
graph.add_node("reviewer", _reviewer)
graph.add_node("tester", _tester)

graph.set_entry_point("coder")
graph.add_edge("coder", "reviewer")       # 固定边
graph.add_edge("reviewer", "tester")       # 固定边

graph.add_conditional_edges("tester", should_continue, {
    "done": END,
    "retry": "coder",
    "max_reached": END,
})
```

**条件路由逻辑**（`should_continue`）：

```python
# graph.py:65-75
def should_continue(state):
    if state["test_results"].get("passed", False):
        return "done"
    if state["iteration"] >= MAX_ITERATIONS:  # MAX_ITERATIONS = 3
        return "max_reached"
    return "retry"
```

**状态定义**（TypedDict，节点间共享）：

```python
# graph.py:33-41
class CodingAgentState(TypedDict):
    task: str                    # 用户任务描述
    messages: list               # 当前对话历史
    generated_code: str          # Coder 产出的代码/摘要
    review_feedback: str         # Reviewer 的反馈
    test_results: dict           # {"passed": bool, "feedback": str}
    iteration: int               # 当前轮次
    trajectory: TrajectoryCollector  # 轨迹收集器
    final_answer: str            # 最终输出
```

### 追问预判

| 追问 | 回答要点 |
|------|---------|
| "Reviewer LGTM 但 Tester FAIL 怎么办？" | Tester 的 FAIL 会触发 retry → 回到 Coder，此时 Coder 收到 test_feedback（测试结果）和 review_feedback（"LGTM"），根据测试失败信息修复代码 |
| "为什么不用手写 while 循环？" | 功能上可以，但 LangGraph 提供：(1) 状态序列化（checkpoint），(2) 可视化，(3) 未来扩展更复杂的分支（如并行 Reviewer） |
| "iteration 在哪里递增？" | 在 `_tester` 包装函数里：`result["iteration"] = result.get("iteration", 0) + 1`（`graph.py:61`） |
| "每轮的 messages 会不会越来越长？" | 每个 Agent 节点会创建独立的 messages 列表（以 system prompt 开头），不会累积之前轮次的完整对话。但会把上一轮的 feedback 作为用户消息传入。 |

## Q: "每个 Agent 内部的 ReAct 循环怎么工作的？"

### 回答

**定位代码**：`agent/nodes/coder.py:78-130`

```python
for step in range(MAX_TOOL_CALLS):       # Coder: 8 次上限
    response = llm.chat(messages, tools)   # LLM 推理
    msg = response["message"]

    if msg.get("tool_calls"):              # LLM 决定调工具
        for tool_call in msg["tool_calls"]:
            name = tool_call["function"]["name"]
            args = tool_call["function"]["arguments"]

            text, is_error = mcp_client.call_tool_parsed(name, args)

            trajectory.record_step(         # 记录到轨迹
                agent="coder", thought=..., action=name,
                action_input=args, observation=text[:500],
                is_error=is_error,
            )

        # 回传工具结果给 LLM
        tool_msgs = llm.make_tool_result_messages(msg, results)
        messages.extend(tool_msgs)

    else:                                  # LLM 不再调工具 → 完成
        state["generated_code"] = msg["content"]
        return state

# 超出上限
state["generated_code"] = "Max tool calls reached"
return state
```

**循环终止条件**（两个）：
1. LLM 返回纯文本（没有 tool_calls）→ 认为任务完成
2. 达到 MAX_TOOL_CALLS 上限 → 强制结束

**每步记录到轨迹**：
- thought：LLM 的推理过程（content 字段）
- action：工具名
- action_input：工具参数
- observation：工具返回（截取前 500 字符）
- is_error：是否失败

### 三个角色的差异

| 角色 | 上限 | 工具权限 | 特殊逻辑 |
|------|------|---------|---------|
| Coder | 8 次 | read_file, write_file, code_search, list_files, execute_code, analyze_code | 后续轮次收到精简的修复指令 |
| Reviewer | 3 次 | read_file, code_search, list_files, analyze_code | 解析 LGTM/NEEDS_REVISION |
| Tester | 5 次 | run_tests, read_file, execute_code | 解析 PASS/FAIL |

### 追问预判

| 追问 | 回答要点 |
|------|---------|
| "如果工具调用失败，Agent 怎么处理？" | 失败信息作为 observation 返回给 LLM，LLM 会看到错误后自主调整策略。比如 pytest 不可用时 Tester 自动改用 `execute_code` 直接运行测试文件。 |
| "ReAct 和 Function Calling 的区别？" | ReAct 是更高层的模式（Thought→Action→Observation 循环），Function Calling 是具体的实现机制。我们用 OpenAI 兼容的 Function Calling API 来实现 ReAct 模式。 |
| "messages 列表怎么构造？" | `[system_prompt, user_task, assistant_tool_call, tool_result, assistant_tool_call, tool_result, ..., assistant_final_response]` |

## Q: "MCP 工具调用的适配怎么做的？"

### 回答

**两层适配**：Schema 格式转换 + 消息格式差异处理

### Schema 适配

**定位代码**：`agent/tool_adapter.py:33-47`

MCP Server 返回的工具 schema：
```json
{"name": "read_file", "description": "...", "inputSchema": {"type": "object", "properties": {...}, "required": [...]}}
```

转成 OpenAI function calling 格式：
```json
{"type": "function", "function": {"name": "read_file", "description": "...", "parameters": {"type": "object", "properties": {...}, "required": [...]}}}
```

关键：`inputSchema` → `parameters`，外层包一个 `{"type": "function", "function": {...}}`。

### 消息格式差异

**定位代码**：`agent/llm.py:49-95` `make_tool_result_messages()`

| 字段 | DeepSeek | Ollama |
|------|----------|--------|
| assistant 消息 | 必须有 `tool_calls[].id` 和 `type: "function"` | 简单 `tool_calls` |
| arguments | 必须是 JSON **字符串** | 可以是 dict |
| tool result | 需要 `tool_call_id` 匹配 | 只需 `{"role": "tool", "content": "..."}` |

**这是坑 5 的来源**：DeepSeek 格式要求比 Ollama 严格得多。

```python
# DeepSeek 分支
hist_msg["tool_calls"].append({
    "id": tc.get("id", _next_call_id()),
    "type": "function",                     # ← 必须有
    "function": {
        "name": tc["function"]["name"],
        "arguments": json.dumps(args),       # ← 必须是字符串
    }
})

# Ollama 分支
msgs = [assistant_msg]
for idx, content in results:
    msgs.append({"role": "tool", "content": content})  # 简单得多
```

### 追问预判

| 追问 | 回答要点 |
|------|---------|
| "为什么不统一用一种格式？" | Ollama 本地模型不完全兼容 OpenAI 格式，强制转换反而容易出错。保持两个分支代码清晰，各自处理自己的格式。 |
| "tool_call_id 是谁生成的？" | DeepSeek API 返回的 assistant 消息里自带 `tc.id`。如果没有（Ollama），用自增 `_next_call_id()` 生成。 |
