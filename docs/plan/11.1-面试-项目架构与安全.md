# 11.1 - 面试问答：项目概述与架构设计

> 面试官会问："介绍一下你的项目"、"哪些是你做的"、"为什么选这个架构"

---

## Q: "介绍一下你这个项目"

### 30 秒版本

> 我基于自己之前写的 C++ MCP Server，搭建了一个多 Agent Coding 系统。用 LangGraph 编排 Coder/Reviewer/Tester 三个 Agent，通过 MCP 协议调用 C++ 实现的 13 个沙箱工具（文件读写、代码执行、测试运行、代码搜索等），自动完成编码任务。同时实现了轨迹收集和 SFT/DPO 训练数据导出，用于 Coding Agent 的数据飞轮。

### 1 分钟版本（加技术栈）

> 在上面的基础上补充：
>
> - 后端用 DeepSeek API（OpenAI 兼容），也支持 Ollama 本地模型双后端切换
> - 安全层：沙箱路径验证（`weakly_canonical` + 前缀检查）、命令白名单、引号感知的危险符号过滤
> - 做了 RAG 代码检索模块（TF-IDF + 余弦相似度），以及 SSE 实时推送 Agent 执行状态
> - 跑了 5 个 Demo，通过轨迹分析发现效率瓶颈，通过 Prompt 工程将步数从 34 步优化到 14 步

### 追问预判

| 追问 | 回答要点 |
|------|---------|
| "MCP 是什么？" | Model Context Protocol，Anthropic 提出的工具调用标准。Server 暴露工具的 JSON Schema，Client 通过 JSON-RPC 2.0 调用。类似 LSP 但面向 AI Agent。 |
| "为什么用 C++ 写 Server？" | 之前学 C++ 的时候已经写了 MCP Server 基础框架，这次在上面扩展 Coding Agent 工具。C++ 的优势是性能好（文件操作、正则匹配）和直接系统调用。 |
| "和 Cursor/Copilot 有什么区别？" | Cursor 是面向人的辅助工具，我这个是全自动的 Agent 系统——给一个任务描述，自动写代码、审查、测试，不需要人介入。重点在于 Agent 协作和训练数据生成。 |

## Q: "哪些是你写的，哪些是 AI 辅助的？"

### 回答模板

> 项目架构设计、技术选型、安全策略是我自己做的。具体实现是我和 AI 协作完成的——我负责定义每个模块的功能需求和接口规范，AI 负责生成初版代码，然后我做 review、debug 和调优。

### 用具体例子支撑（必须说 2-3 个）

**例子 1：引号感知解析器**

> `execute_code_tool.cpp` 的安全检查，最初的分号过滤会误杀 `python3 -c "a=1; b=2"` 这种合法命令。这个问题是我在分析 Demo 轨迹时发现的——工具成功率只有 87%，排查发现大量 execute_code 被分号过滤拦截。是我设计了"只检查引号外"的方案（状态机追踪引号状态），然后让 AI 实现 `extract_unquoted_parts()` 函数。

**定位代码**：`src/server/builtin/tools/execute_code_tool.cpp:26-55`

**例子 2：Ollama 502 排查**

> Ollama 调用一直 502，AI 尝试了清环境变量、改 SDK 参数、传自定义 httpx.Client，三次都失败了。是我用 `curl --noproxy localhost` 测试发现 Ollama 本身正常，排查出是 `ollama` Python SDK 内部的 httpx 库走了系统代理。我决定弃用 SDK，改用 requests 库直接调 REST API。

**定位代码**：`agent/llm.py:156-189`

**例子 3：write_file 安全漏洞**

> Code Review 时我发现 `write_file` 没有做路径验证，Agent 可以写任意文件（路径穿越漏洞）。`read_file` 有沙箱检查但 `write_file` 漏了，我让 AI 补上了和 `read_file` 一样的 `weakly_canonical` + 前缀检查逻辑。

**定位代码**：`src/server/builtin/tools/write_file_tool.cpp:35-50`

### 追问预判

| 追问 | 回答要点 |
|------|---------|
| "AI 写的代码你能解释吗？" | 能，每一行都清楚。（然后随便指一个文件讲） |

## Q: "为什么选 LangGraph？为什么三个 Agent？"

### 回答

> **LangGraph vs AutoGen vs CrewAI**：
>
> - AutoGen 偏向对话式多 Agent，适合讨论型任务，但对工具调用的控制不够精细
> - CrewAI 封装太重，自定义工具和状态管理不灵活
> - LangGraph 是状态图模型，天然支持：
>   - **条件路由**：测试通过 → 结束，失败 → 回到 Coder 重做
>   - **状态序列化**：可以 checkpoint 中间状态，断点恢复
>   - **可视化**：状态图可以直接画出来，容易理解和调试
>
> **为什么三个 Agent**：
>
> - Coder：写代码，有 write_file、execute_code 权限
> - Reviewer：审代码，只有 read_file 权限（不能改代码，防止审查者自己修改）
> - Tester：跑测试，有 run_tests、execute_code 权限
>
> 三个 Agent 的工具权限不同，这是通过 `tool_adapter.py:73-89` 的角色过滤实现的。如果一个 Agent 同时写代码和审查，会有"自己审自己"的问题。

**定位代码**：
- 状态图：`agent/graph.py:44-103`
- 角色过滤：`agent/tool_adapter.py:73-89`
- 条件路由：`agent/graph.py:65-75` `should_continue()`

### 追问预判

| 追问 | 回答要点 |
|------|---------|
| "最多迭代几轮？" | 3 轮（`config.py:MAX_ITERATIONS=3`），防止死循环。实际多数任务 1 轮通过。 |
| "如果 3 轮还没通过呢？" | 标记为 FAIL，导出的轨迹作为 DPO 的 rejected 数据。 |
| "状态里传了什么？" | TypedDict：task、messages、generated_code、review_feedback、test_results、iteration、trajectory（详见 `graph.py:33-41`） |
