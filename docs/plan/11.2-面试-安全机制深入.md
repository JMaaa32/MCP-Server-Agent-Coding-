# 11.2 - 面试问答：安全机制深入

> 安全是这个项目的亮点，面试官很可能深入追问。

---

## Q: "解释一下你的沙箱安全机制"

### 回答

> 三层防御：路径沙箱 + 命令白名单 + 危险符号过滤。

### 第一层：路径沙箱

**定位代码**：`src/server/builtin/tools/read_file_tool.cpp:20-44`

```cpp
// 1. 用户传入相对路径
fs::path target = raw_path;
if (target.is_relative()) {
    target = sandbox_root / target;  // 基于 workspace/ 解析
}

// 2. 规范化（解析 .. 和 symlink）
fs::path canonical_path = fs::weakly_canonical(target, ec);

// 3. 前缀检查
auto sandbox_str = sandbox_root.string();
auto target_str = canonical_path.string();
if (target_str.substr(0, sandbox_str.size()) != sandbox_str) {
    return error;  // 路径穿越，拒绝
}
```

**攻击示例**：
- `../../etc/passwd` → `weakly_canonical` 解析成 `/etc/passwd` → 不以 `/path/to/workspace` 开头 → 拒绝
- `workspace/../../../etc/passwd` → 同理，规范化后会解析出真实路径 → 拒绝

**为什么用 `weakly_canonical` 而不是 `canonical`**：
- `canonical()` 要求路径上所有组件必须**实际存在**，否则抛异常
- `write_file` 的目标文件通常还不存在（创建新文件）
- `weakly_canonical()` 会尽可能解析已存在的部分，不存在的保留原样

### 第二层：命令白名单

**定位代码**：`src/server/builtin/tools/execute_code_tool.cpp:14-22`

```cpp
static const std::vector<std::string> ALLOWED_PREFIXES = {
    "python3", "python", "node",
    "g++", "gcc", "clang++", "clang",
    "ls", "cat", "grep", "rg",
    "head", "tail", "wc", "find",
    "echo", "pwd", "stat", "file",
    "sort", "uniq", "diff", "tr"
};
```

检查逻辑：命令去除前导空格后，检查是否以白名单中的某个前缀开头。

### 第三层：引号感知的危险符号过滤

**定位代码**：`execute_code_tool.cpp:26-72`

**问题**：`python3 -c "import os; print(os.getcwd())"` 中的分号在引号内，对 shell 是安全的。但原版检测把整个命令当字符串搜索 `;`，导致误杀。

**解决**：实现状态机 `extract_unquoted_parts()`，只提取引号外的部分来检查。

```cpp
static std::string extract_unquoted_parts(const std::string& cmd) {
    std::string result;
    bool in_single = false, in_double = false, escaped = false;
    for (size_t i = 0; i < cmd.size(); ++i) {
        char c = cmd[i];
        if (escaped) { escaped = false; continue; }
        if (c == '\\') { escaped = true; continue; }
        if (c == '\'' && !in_double) { in_single = !in_single; continue; }
        if (c == '"' && !in_single) { in_double = !in_double; continue; }
        if (!in_single && !in_double) { result += c; }
    }
    return result;
}
```

**状态机有 4 个状态**：普通、单引号内、双引号内、转义。

**验证**：
```
✅ python3 -c "import os; print(os.getcwd())"  → 引号内分号，允许
✅ python3 -c 'a=1; b=2'                        → 单引号内分号，允许
❌ python3 test.py; rm -rf /                     → 引号外分号，拦截
❌ python3 test.py && cat /etc/passwd             → 引号外 &&，拦截
```

### 追问预判

| 追问 | 回答要点 |
|------|---------|
| "能不能绕过白名单？" | 白名单检查的是前缀，`python3` 开头就允许。理论上 `python3 -c "import subprocess; subprocess.run('rm -rf /')"` 可以绕过。但这需要 LLM 被 prompt injection，实际场景中概率低。如果要加强，可以在 Python 层面限制可用模块。 |
| "symlink 攻击呢？" | `weakly_canonical` 会解析 symlink，所以 `ln -s /etc/passwd workspace/link` 后读 `link` 会被解析成 `/etc/passwd` → 拒绝。前提是 workspace 内不能有指向外部的 symlink（Server 启动时应检查）。 |
| "为什么不用 chroot？" | chroot 需要 root 权限，且要在 chroot 环境里准备完整的工具链（python3, node 等）。沙箱前缀检查是用户态方案，部署简单。生产环境可以考虑 Docker 容器。 |
| "extract_unquoted_parts 有没有 bug？" | 不处理 here-doc（`<<EOF`）和 backtick 嵌套引号的情况。但 ALLOWED_PREFIXES 限制了命令只能是 python3/node 等开发工具，实际不会用到 here-doc。 |

## Q: "遇到过什么安全相关的问题？"

### write_file 路径穿越漏洞

> Code Review 时发现 `read_file` 有沙箱验证但 `write_file` 没有。如果 LLM 被 prompt injection（比如读到一个包含恶意指令的文件），它可以调 `write_file(path="../../.bashrc", content="malicious")` 写入任意文件。

**修复**：给 `write_file_tool.cpp:35-50` 加上同样的 `weakly_canonical` + 前缀检查。

### 分号注入误杀

> 不算安全漏洞，但是安全策略太严导致可用性问题。分号检测把引号内的安全分号也拦截了，导致 Agent 执行 Python 代码时 13% 的工具调用被误拒。

**修复**：引号感知解析器（上面详述）。

### 面试话术

> 安全设计要平衡严格性和可用性。太松了有路径穿越和命令注入风险，太严了会误杀合法操作。我们通过白名单（限制攻击面）+ 引号感知过滤（减少误杀）+ 路径规范化（防穿越）三层组合，在安全和可用之间取得平衡。
